# Release and Immutability Policy

An annotated semantic-version tag identifies one exact reviewed commit. A published tag is never moved, and published assets are never silently replaced. Corrective work uses a new patch release; compatible feature work uses a new minor release; methodology-breaking changes require an appropriate major-version decision.

Release manifests bind an implementation/configuration commit. Mechanical manifest regeneration follows that boundary, and release evidence may follow the manifest commit. Strict release-candidate validation always receives an explicit exact candidate SHA. `scripts/validate_release_provenance.py --candidate SHA` requires exact implementation-to-manifest-to-candidate ancestry, permits only `manifests/` before manifest capture and only `audit/release_v1/` afterward, and examines every touched path even if a later commit reverted it. Runtime, configuration, workflow, dependency, documentation, or methodology changes after manifest capture therefore remain release-candidate failures.

Published-release validation is a separate operation. `scripts/validate_release_provenance.py --tagged-release` resolves the release name from the provenance record, requires the recorded tag to exist as an annotated tag, verifies its recorded tag-object and peeled-target SHAs, and applies the same strict closure to the peeled target. Current `main` and pull-request commits are reported separately and never become the v1.0.0 release candidate.

Current-main CI builds and tests the evolving checkout normally. Immutable v1.0.0 reconstruction runs separately through `scripts/reconstruct_tagged_release.py`, which validates the tag, creates a clean detached temporary worktree at the exact peeled target, installs the hash-locked validation environment from that worktree, reconstructs the complete suite in a separate build/output area, runs the manifest and output validators, and removes the worktree without changing the current checkout. No pull-request-supplied ref selects the release source.

Platform binary archives are published only after build, test, extraction, version, help, configuration, representative synthetic experiment, result-validation, data-boundary, secret/path-scan, SBOM, and checksum gates pass on the named target runner. Release source and reproducibility archives contain project-owned synthetic fixtures, not third-party market data.

The GitHub Release notes and checksums are part of the release record. Any discovered post-release defect is documented and corrected through a new version rather than mutable replacement.
